## 一、产品概述

### 1.1 背景与问题

交易所上币是高影响力决策。现有流程：

- 分析师手动查 CoinGecko、GitHub、社区数据
- 无结构化评分标准，结论主观
- 单次调研耗时 **2-3 个工作日**
- 不同分析师结论差异大，缺乏一致性

**核心问题**：上币决策依赖人工经验，效率低、标准不统一、覆盖面有限。

### 1.2 产品定位

> 输入一个 Token 名称，3 分钟内输出结构化上币评估报告。

Token Lens 是一个 **AI 驱动的上币评估工具**，自动聚合多维度公开数据，由 Claude 进行综合分析，输出可执行的推荐结论。

### 1.3 目标用户

| 用户 | 使用场景 |
|------|----------|
| 上币运营团队 | 快速筛选待评估项目，生成初步评估报告 |
| 业务负责人 | 路演/决策前的参考依据 |
| 研究分析师 | 替代重复性数据收集工作 |

### 1.4 核心价值

| 旧方式 | Token Lens |
|--------|-----------|
| 2-3 天人工调研 | 3 分钟自动完成 |
| 主观判断 | 结构化评分 + AI 分析 |
| 数据散落各处 | 统一报告，一目了然 |
| 个人经验依赖 | 可复用、可交接的标准流程 |

## 四、数据源

### 4.1 CoinGecko API（唯一数据源）

> **设计原则**：单一数据源，零额外依赖。CoinGecko 免费接口已包含市场、社区、开发者活跃度和交易所上架信息，无需额外调用 GitHub API 或 DeFiLlama。

#### 步骤一：搜索 Token ID

**接口**：`GET https://api.coingecko.com/api/v3/search?query={name}`

**用途**：用户输入 Token 名称（如 "SUI"）后，通过此接口获取 CoinGecko 内部 coin_id，再用于后续详情查询。

**返回字段**：`coins[].id`（如 `"sui"`）、`coins[].name`、`coins[].symbol`，展示给用户确认选择。

#### 步骤二：获取详情数据

**接口**：`GET https://api.coingecko.com/api/v3/coins/{id}?tickers=true`

**获取字段**：

| 字段 | 用途 |
|------|------|
| `market_data.market_cap.usd` | 市值 |
| `market_data.total_volume.usd` | 24h 交易量 |
| `market_data.price_change_percentage_30d` | 30日价格波动率 |
| `market_data.market_cap_rank` | 市值排名 |
| `community_data.twitter_followers` | Twitter 粉丝数 |
| `community_data.telegram_channel_user_count` | Telegram 成员数 |
| `developer_data.stars` | GitHub Star 数 |
| `developer_data.forks` | Fork 数 |
| `developer_data.commit_count_4_weeks` | 近4周提交数 |
| `developer_data.last_4_weeks_commit_activity_series` | 每周提交趋势（判断活跃度走向） |
| `tickers[].market.name` | 已上线的交易所名称（传给 Claude 前需裁剪，见下方说明） |

**限制**：免费版 10-30 req/min，无需 API Key（有 Key 额度更高）

> **tickers 裁剪规则**：热门 Token 的 tickers 数组可达数百至上千条，直接传给 Claude 会导致 prompt 超长。代码中只保留主流交易所名称后再传入：
>
> ```python
> MAJOR_EXCHANGES = {"Binance", "OKX", "Bybit", "Coinbase", "Kraken", "Bitget", "Gate.io", "BYDFi"}
>
> def extract_listed_exchanges(tickers: list) -> tuple[list[str], int]:
>     names = {t["market"]["name"] for t in tickers if t.get("market")}
>     major = sorted(names & MAJOR_EXCHANGES)
>     others_count = len(names) - len(major)
>     return major, others_count  # 传给 Claude: major所列表 + "及其他N家"
> ```

## 五、评分体系

### 5.1 维度说明

| 维度 | 权重 | 满分 | 数据来源 |
|------|------|------|----------|
| 市场规模 | 30% | 30 | CoinGecko market_data |
| 社区活跃度 | 20% | 20 | CoinGecko community_data |
| 技术实力 | 20% | 20 | CoinGecko developer_data |
| 竞争位置 | 15% | 15 | Claude 分析（基于 tickers 实时数据） |
| 风险信号 | 15% | 15 | Claude 基于全量数据判断 |

### 5.2 各维度评分规则

#### 市场规模（满分 30）

| 指标 | 规则 | 分值 |
|------|------|------|
| 市值排名 | Top 20: 15分 / Top 100: 10分 / Top 500: 6分 / 其他: 3分 | 15 |
| 24h 交易量 | >$500M: 10分 / >$100M: 7分 / >$10M: 5分 / <$10M: 2分 | 10 |
| 30日波动率 | <30%: 5分 / 30-60%: 3分 / >60%: 1分 | 5 |

#### 社区活跃度（满分 20）

| 指标 | 规则 | 分值 | 异常检测 |
|------|------|------|---------|
| Twitter 粉丝数 | >500k: 12分 / >100k: 9分 / >10k: 5分 / <10k: 2分 / 无数据: 0分 | 12 | 粉丝数 >500k 且 24h 交易量 <$1M，标记"高粉丝/低流动性异常" |
| Telegram 成员数 | >100k: 8分 / >10k: 5分 / >1k: 3分 / 无: 0分 | 8 | — |

> 说明：移除原 GitHub 近4周提交分项（该数据已在"技术实力"维度计分，重复使用会导致双重计数）。

#### 技术实力（满分 20）

> **数据来源**：全部从 CoinGecko developer_data 获取，无需单独调用 GitHub API。

| 指标 | 规则 | 分值 |
|------|------|------|
| GitHub Star 数 | >10k: 10分 / >1k: 7分 / >100: 4分 / <100或无: 1分 | 10 |
| 近4周提交活跃度 | >50次: 10分 / >10次: 7分 / >0次: 3分 / 0次: 0分 | 10 |

> 说明：移除原 TVL 维度（DeFiLlama 覆盖率低，大量 Token 无数据，导致评分失真）。Star 数满分从 8 调整为 10，提交活跃度满分从 5 调整为 10，总分不变。

#### 竞争位置（满分 15）

**由 Claude 基于以下结构化数据判断**（非 Claude 记忆，全部来自 CoinGecko tickers 实时数据）：
- 已上线交易所列表（tickers 字段提取）
- 是否已在 Binance/OKX/Bybit/BYDFi 等主流所上线（基于实时数据，不依赖 Claude 训练集）
- 市值排名在同赛道中的位置

**输出**：0-15 的整数分，附 1-2 句说明

#### 风险信号（满分 15，从满分扣减）

风险扣分分为两类：**规则引擎直接判断**（有明确数据依据）和 **Claude 综合判断**（需要语义理解）。

**规则引擎扣分**（在 scorer.py 中实现，不依赖 Claude）：

| 风险类型 | 扣分 | 判断依据 |
|----------|------|---------|
| 代码长期停止更新 | -5 | `commit_count_4_weeks == 0` 且 `developer_data` 非空 |
| 无 GitHub 开源代码 | -3 | `developer_data.stars` 为 null 且 `links.repos_url.github` 为空 |
| 近期价格剧烈下跌 | -3 | `price_change_percentage_30d < -50%` |
| 社区与市值严重不匹配 | -2 | 市值排名 Top 100 但 Twitter 粉丝 <10k（且粉丝数据非 null）|

**Claude 综合判断扣分**（需语义分析，传入全量数据后由 Claude 决定）：

| 风险类型 | 最大扣分 | 说明 |
|----------|---------|------|
| 项目定位模糊/赛道过度拥挤 | -2 | Claude 基于项目描述和市值排名判断 |

> 说明：原"交易量异常暴增"风险项已移除。该判断需要历史数据对比，当前输入中只有单点 24h 交易量，Claude 无法可靠检测，强行保留会导致 Claude 随机输出或编造结论。

**基础分 15，扣减后最低 0 分**

### 5.3 推荐结论

| 总分 | 结论 | 颜色 |
|------|------|------|
| ≥ 75 | 强烈推荐上币 | 绿色 |
| 55-74 | 建议观望，重点关注风险点 | 黄色 |
| < 55 | 不建议上币 | 红色 |

## 六、Claude 分析模块

### 6.1 输入给 Claude 的内容

```
结构化数据摘要（JSON）：
- Token 基础信息（名称、市值、交易量、市值排名、30日价格变化）
- 各项原始数据指标（社区、开发者数据）
- 规则评分结果（前3个维度分数 + 规则引擎已识别的风险及扣分）
- 已上线主流交易所列表（经裁剪，仅主流所）
  示例：{"major": ["Binance", "OKX", "Bybit"], "other_count": 42}
- 是否已在 BYDFi 上线（布尔值）

分析任务：
1. 基于已上线交易所列表，判断该项目竞争位置，给出 0-15 分
2. 在规则引擎扣分基础上，识别语义层面的额外风险（最多扣2分），给出最终风险得分
3. 输出 3 条推荐/不推荐核心理由
4. 输出 2 条最需关注的风险点
5. 判断 BYDFi 跟进上币的紧迫性（高/中/低）及理由
6. 一句话总结推荐结论

输出格式：严格 JSON，不加 markdown 代码块，不加解释性文字
```

### 6.2 Claude 输出格式（结构化 JSON）

```json
{
  "competitive_score": 12,
  "competitive_reason": "已在 Binance、OKX、Bybit 上线，头部所均有覆盖，流动性充足",
  "risk_score": 10,
  "risk_extra_deduction": 2,
  "risk_extra_reason": "项目定位与同赛道多个头部项目高度重叠，差异化不明显",
  "top_reasons": [
    "市值排名 Top 50，交易量充足，流动性风险低",
    "Twitter 粉丝超 80 万，社区基础扎实",
    "已在3家头部交易所上线，市场认可度高"
  ],
  "top_risks": [
    "近期开发活跃度下降，需关注团队是否仍在积极维护",
    "赛道竞争激烈，差异化优势需进一步验证"
  ],
  "bydfi_urgency": "高",
  "bydfi_urgency_reason": "竞品已全面覆盖，BYDFi 尚未上线，存在用户流失风险，建议尽快跟进",
  "summary": "综合评估建议上币，流动性和社区基础良好，需关注开发活跃度。"
}
```

### 6.3 Claude 输出解析容错

Claude 有时会在 JSON 外包裹 markdown 代码块或输出格式不规范。`claude_analyzer.py` 必须实现以下容错：

```python
import re, json

def parse_claude_response(text: str) -> dict:
    # 剥掉可能存在的 markdown 代码块
    match = re.search(r"```(?:json)?\s*(\{.*?\})\s*```", text, re.DOTALL)
    raw = match.group(1) if match else text.strip()
    try:
        return json.loads(raw)
    except json.JSONDecodeError:
        # 降级：AI 分析部分返回中位分，在报告中标注"AI 分析异常"
        return {
            "competitive_score": 8,
            "risk_score": 8,
            "risk_extra_deduction": 0,
            "top_reasons": ["AI 分析暂不可用，以下为规则评分结果"],
            "top_risks": [],
            "bydfi_urgency": "中",
            "bydfi_urgency_reason": "AI 分析异常，建议人工核查",
            "summary": "AI 分析模块异常，当前展示规则评分结果。",
            "parse_error": True
        }
```

## 九、异常处理规范

| 场景 | 处理方式 |
|------|----------|
| CoinGecko 返回 404 | 提示 Token 名称错误，展示搜索建议 |
| CoinGecko 限流（429） | 立即提示"请求频率过高，请稍后重试"，不静默等待（等待会冻结 UI） |
| developer_data 字段为 null | 技术评分相关项计 0 分，报告标注"无开源数据" |
| tickers 字段为空 | 竞争位置分析不含交易所列表，Claude 基于市值排名判断 |
| community_data.twitter_followers 为 null | 社区分对应项计 0 分，标注"粉丝数据不可用" |
| Claude API 超时 | 重试一次，仍失败则展示规则评分，跳过 AI 分析部分 |
| 网络完全断开 | 全局错误提示 |

## 十、验收标准

### 路演前必须满足

- [ ] 输入任意主流 Token 名称，能在 3 分钟内出完整报告
- [ ] 5 个维度均有评分，总分正常计算
- [ ] 雷达图正常渲染
- [ ] Claude 分析文字有实质内容，不是模板废话
- [ ] 最终推荐结论与数据逻辑一致
- [ ] 换一个 Token 还能跑（不硬编码数据）
- [ ] 有明确的加载状态，不会白屏卡死

### 路演演示准备

准备 3 个 Token，覆盖三种结论：

| Token | 预期结论 | 用途 |
|-------|----------|------|
| BTC 或 ETH | 强烈推荐 | 展示高分案例 |
| 中等项目（如 OP、ARB） | 建议观望 | 展示区分度 |
| 冷门小币 | 不建议 | 展示风险识别能力 |
