## 一、产品概述

### 1.1 背景与问题

交易所上币是高影响力决策。现有流程：

- 分析师手动查 CoinGecko、GitHub、社区数据
- 无结构化评分标准，结论主观
- 单次调研耗时 **2-3 个工作日**
- 不同分析师结论差异大，缺乏一致性

**核心问题**：上币决策依赖人工经验，效率低、标准不统一、覆盖面有限。

### 1.2 产品定位

> 输入一个 Token 名称，3 分钟内输出结构化上币评估报告。

Token Lens 是一个 **AI 驱动的上币评估工具**，自动聚合多维度公开数据，由 Claude 进行综合分析，输出可执行的推荐结论。

### 1.3 目标用户

| 用户 | 使用场景 |
|------|----------|
| 上币运营团队 | 快速筛选待评估项目，生成初步评估报告 |
| 业务负责人 | 路演/决策前的参考依据 |
| 研究分析师 | 替代重复性数据收集工作 |

### 1.4 核心价值

| 旧方式 | Token Lens |
|--------|-----------|
| 2-3 天人工调研 | 3 分钟自动完成 |
| 主观判断 | 结构化评分 + AI 分析 |
| 数据散落各处 | 统一报告，一目了然 |
| 个人经验依赖 | 可复用、可交接的标准流程 |

## 二、功能范围

### 2.1 核心功能（MVP 必须交付）

| 功能 | 描述 | 优先级 |
|------|------|--------|
| Token 搜索 | 输入 Token 名称，模糊匹配候选列表 | P0 |
| 数据自动采集 | 调用 CoinGecko / DeFiLlama / GitHub API | P0 |
| 多维度评分 | 5 个维度自动计分，输出总分 | P0 |
| Claude AI 分析 | 综合数据生成推荐理由 + 风险点 | P0 |
| 评估报告展示 | 雷达图 + 评分条 + 文字分析 | P0 |
| 最终推荐结论 | 强烈推荐 / 建议观望 / 不建议 | P0 |

### 2.2 非核心功能（本次不做）

- 历史记录持久化（数据库）
- 用户登录/权限管理
- PDF 导出
- 多 Token 横向对比
- 合约地址直接输入
- 邮件/消息通知

## 三、用户流程

```
用户打开系统
    ↓
搜索框输入 Token 名称（如 "SUI"）
    ↓
系统返回候选列表，用户确认选择
    ↓
系统展示"数据采集中..."进度状态
    ↓
    ├── 调用 CoinGecko API（市场 + 社区数据）
    ├── 调用 DeFiLlama API（TVL 数据）
    └── 调用 GitHub API（代码活跃度）
    ↓
数据传入评分引擎，计算 5 个维度得分
    ↓
数据 + 评分结果传入 Claude，生成分析文字
    ↓
展示完整评估报告
    ├── 基础信息卡片
    ├── 5维度评分条
    ├── 雷达图
    ├── Claude 分析文字
    └── 最终推荐结论
```

## 四、数据源

### 4.1 CoinGecko API（主力数据源）

**接口**：`GET https://api.coingecko.com/api/v3/coins/{id}`

**获取字段**：

| 字段 | 用途 |
|------|------|
| `market_data.market_cap.usd` | 市值 |
| `market_data.total_volume.usd` | 24h 交易量 |
| `market_data.price_change_percentage_30d` | 30日价格波动率 |
| `market_data.market_cap_rank` | 市值排名 |
| `community_data.twitter_followers` | Twitter 粉丝数 |
| `community_data.telegram_channel_user_count` | Telegram 成员数 |
| `developer_data.stars` | GitHub Star 数 |
| `developer_data.commit_count_4_weeks` | 近4周提交数 |
| `links.repos_url.github` | GitHub 仓库地址 |

**限制**：免费版 10-30 req/min，无需 API Key（有 Key 额度更高）

### 4.2 DeFiLlama API

**接口**：`GET https://api.llama.fi/protocol/{slug}`

**获取字段**：

| 字段 | 用途 |
|------|------|
| `currentChainTvls` | 各链 TVL |
| `tvl`（数组最后一条） | 当前总 TVL |

**限制**：完全免费，无需 Key，无速率限制

### 4.3 GitHub API（可选增强）

**接口**：`GET https://api.github.com/repos/{owner}/{repo}`

**获取字段**：

| 字段 | 用途 |
|------|------|
| `stargazers_count` | Star 数（与 CoinGecko 互补验证） |
| `forks_count` | Fork 数 |
| `open_issues_count` | 开放 Issue 数 |
| `pushed_at` | 最近推送时间 |

**限制**：未认证 60 req/h，加 Token 后 5000 req/h

> 注：GitHub 仓库地址从 CoinGecko 的 `links.repos_url.github` 字段获取，无需单独维护。

## 五、评分体系

### 5.1 维度说明

| 维度 | 权重 | 满分 | 数据来源 |
|------|------|------|----------|
| 市场规模 | 30% | 30 | CoinGecko |
| 社区活跃度 | 20% | 20 | CoinGecko 社区字段 |
| 技术实力 | 20% | 20 | GitHub + DeFiLlama |
| 竞争位置 | 15% | 15 | Claude 基于市值排名分析 |
| 风险信号 | 15% | 15 | Claude 基于全量数据判断 |

### 5.2 各维度评分规则

#### 市场规模（满分 30）

| 指标 | 规则 | 分值 |
|------|------|------|
| 市值排名 | Top 20: 15分 / Top 100: 10分 / Top 500: 6分 / 其他: 3分 | 15 |
| 24h 交易量 | >$500M: 10分 / >$100M: 7分 / >$10M: 5分 / <$10M: 2分 | 10 |
| 30日波动率 | <30%: 5分 / 30-60%: 3分 / >60%: 1分 | 5 |

#### 社区活跃度（满分 20）

| 指标 | 规则 | 分值 |
|------|------|------|
| Twitter 粉丝数 | >500k: 10分 / >100k: 7分 / >10k: 4分 / <10k: 2分 | 10 |
| Telegram 成员数 | >100k: 5分 / >10k: 3分 / >1k: 2分 / 无: 0分 | 5 |
| GitHub 近4周提交 | >50: 5分 / >10: 3分 / >0: 1分 / 0: 0分 | 5 |

#### 技术实力（满分 20）

| 指标 | 规则 | 分值 |
|------|------|------|
| GitHub Star 数 | >10k: 8分 / >1k: 5分 / >100: 3分 / <100或无: 1分 | 8 |
| 代码最近更新 | 30天内: 7分 / 90天内: 4分 / 180天内: 2分 / 更久: 0分 | 7 |
| TVL 规模 | >$1B: 5分 / >$100M: 3分 / >$10M: 2分 / 无/低: 0分 | 5 |

#### 竞争位置（满分 15）

**由 Claude 基于以下信息综合判断**：
- 市值排名在同赛道中的位置
- 是否已在 Binance/OKX 等头部交易所上线
- 赛道整体热度与项目差异化

**输出**：0-15 的整数分，附 1-2 句说明

#### 风险信号（满分 15，从满分扣减）

**由 Claude 基于全量数据识别以下风险**，每项风险扣对应分数：

| 风险类型 | 扣分 |
|----------|------|
| 代码长期停止更新（>6个月无提交） | -5 |
| 交易量异常（单日暴增 >500%，无明显催化剂） | -5 |
| 社区数据与市值严重不匹配（高市值低社区）| -3 |
| 无 GitHub 开源代码 | -3 |
| 近期价格剧烈下跌（>50%，无市场性原因）| -3 |

**基础分 15，Claude 识别风险后扣减，最低 0 分**

### 5.3 推荐结论

| 总分 | 结论 | 颜色 |
|------|------|------|
| ≥ 75 | 强烈推荐上币 | 绿色 |
| 55-74 | 建议观望，重点关注风险点 | 黄色 |
| < 55 | 不建议上币 | 红色 |

## 六、Claude 分析模块

### 6.1 输入给 Claude 的内容

```
结构化数据摘要（JSON）：
- Token 基础信息
- 各项原始数据指标
- 规则评分结果（前3个维度的分数）

分析任务：
1. 判断该项目在赛道中的竞争位置，给出 0-15 分
2. 识别风险信号，给出 0-15 分（从15开始扣）
3. 输出 3 条推荐/不推荐核心理由
4. 输出 2 条最需关注的风险点
5. 一句话总结推荐结论
```

### 6.2 Claude 输出格式（结构化 JSON）

```json
{
  "competitive_score": 12,
  "competitive_reason": "该项目在 Layer2 赛道中排名前3，已在 Binance、OKX 上线，流动性充足",
  "risk_score": 10,
  "risk_deductions": [
    {"risk": "近30天代码提交数偏低", "deduction": 3, "detail": "仅4次提交，低于同类项目平均水平"}
  ],
  "top_reasons": [
    "市值排名 Top 50，交易量充足，流动性风险低",
    "Twitter 粉丝超 80 万，社区基础扎实",
    "已通过 Certik 公开审计（根据 CoinGecko 审计标签）"
  ],
  "top_risks": [
    "近期开发活跃度下降，需关注团队是否仍在积极维护",
    "赛道竞争激烈，差异化优势需进一步验证"
  ],
  "summary": "综合评估建议上币，流动性和社区基础良好，需关注开发活跃度。"
}
```

## 九、异常处理规范

| 场景 | 处理方式 |
|------|----------|
| CoinGecko 返回 404 | 提示 Token 名称错误，展示搜索建议 |
| CoinGecko 限流（429） | 等待 10s 后重试一次，失败则提示 |
| DeFiLlama 无数据 | TVL 显示"N/A"，该分项计 0 分 |
| GitHub 仓库不存在 | 技术评分相关项计最低分 |
| Claude API 超时 | 重试一次，仍失败则展示规则评分，跳过 AI 分析部分 |
| 网络完全断开 | 全局错误提示 |

## 十、验收标准

### 路演前必须满足

- [ ] 输入任意主流 Token 名称，能在 3 分钟内出完整报告
- [ ] 5 个维度均有评分，总分正常计算
- [ ] 雷达图正常渲染
- [ ] Claude 分析文字有实质内容，不是模板废话
- [ ] 最终推荐结论与数据逻辑一致
- [ ] 换一个 Token 还能跑（不硬编码数据）
- [ ] 有明确的加载状态，不会白屏卡死

### 路演演示准备

准备 3 个 Token，覆盖三种结论：

| Token | 预期结论 | 用途 |
|-------|----------|------|
| BTC 或 ETH | 强烈推荐 | 展示高分案例 |
| 中等项目（如 OP、ARB） | 建议观望 | 展示区分度 |
| 冷门小币 | 不建议 | 展示风险识别能力 |
