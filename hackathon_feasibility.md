# BYDFi AI Reforge Hackathon — 参赛方案可行性分析

> 评估日期：2026-03-27
> 比赛时间：3月30日 - 4月6日（7天）
> 核心要求：Claude 为主力引擎，产出能跑的端到端系统

---

## 评审权重提醒

| 维度 | 权重 |
|------|------|
| 认知突破度 | 25% |
| 创新价值 | 25% |

## 方案一：竞品体验差距自动诊断系统

### 命题方向
全站体验优化（方向 1）

### 目标段位
**Lv.3（+25%）** — 掀自己团队的桌子，指出当前产品不足

### 问题陈述
BYDFi 产品团队做体验优化，靠的是主观感知或零散用户反馈。没有系统化的方法来衡量"我们和 Binance/OKX 的差距到底在哪、差多少"。

### 系统设计

```
输入：BYDFi 网站 URL + 竞品列表（Binance / OKX / Bybit）
  ↓
爬取各平台关键页面（首页、注册、现货/合约交易页、帮助中心）
  ↓
Claude 多维度逐项对比分析：
  - 注册/登录流程步骤数 & 摩擦点
  - 首屏信息密度 & 可读性
  - 核心交易功能完整性
  - 文案质量（清晰度、行动引导）
  - 移动端适配体验
  ↓
输出：结构化差距报告 + 优先级改进 Backlog（附截图对比）
```

### 技术栈
- Python + Playwright（爬取页面截图 + 文本）
- Claude API（分析 + 对比 + 生成报告）
- Markdown / HTML 报告输出

### 可行性评估
- **开发难度**：中等（爬虫 + Claude 调用，无复杂后端）
- **数据来源**：全部是竞品公开信息（符合安全红线）
- **7天能完成**：可以，核心流程 3天可跑通，剩余时间优化报告质量
- **路演亮点**：现场输入一个竞品 URL，实时跑出对比报告，直接让评委看到结论

### 风险
- 竞品页面反爬可能需要绕过（用 Playwright headless + 延时）
- 报告质量依赖 prompt 设计质量

## 方案二：加密市场竞情自动监控系统

### 命题方向
工作效率提升（方向 4）

### 目标段位
**Lv.2（+15%）** — 解决团队实际重复性工作

### 问题陈述
运营/市场团队每天手动刷竞品公告、新上币、活动、费率变化，耗时且容易漏信息。没有一个系统自动聚合并解读这些信号。

### 系统设计

```
输入：竞品列表（Binance、OKX、Bybit、Bitget 等）
  ↓
定时爬取：
  - 官方公告页（新上币、下线、规则变更）
  - 活动页（利率、奖励活动）
  - 费率页（手续费变化）
  ↓
Claude 分析每条信息：
  - 重要性评分（高/中/低）
  - 对 BYDFi 的影响判断
  - 是否需要跟进动作
  ↓
输出：每日竞情简报（按重要性排序）+ 高优先告警推送
```

### 技术栈
- Python + BeautifulSoup / Playwright
- Claude API（重要性判断 + 影响分析）
- 输出为 Markdown 日报 / 可选接入飞书 webhook

### 可行性评估
- **开发难度**：低（信息爬取成熟方案多）
- **实际价值**：高，团队每天都用得上
- **7天能完成**：非常可以，2天跑通核心，5天完善覆盖面
- **路演亮点**：展示昨天的竞情日报，和人工整理版本对比效率

### 风险
- 段位只有 Lv.2，竞争力相对弱
- 建议与其他方向结合以提高影响层级

## 方案三：客服工单 AI 诊断与知识库自动更新系统

### 命题方向
内部流程重塑（方向 3）

### 目标段位
**Lv.4（+30%）** — 掀其他部门的桌子（技术视角，给客服部门提优化建议）

### 问题陈述
客服团队处理大量重复工单，但这些数据从未被系统性分析过：哪些问题最高频？根因是产品 Bug 还是文档不清晰？哪些问题可以用 FAQ 拦截？

### 系统设计

```
输入：历史客服工单数据（CSV / 脱敏文本，可用模拟数据演示）
  ↓
Claude 多轮分析：
  Step 1 - 分类聚类（提款问题 / KYC问题 / 交易问题 / 账户问题...）
  Step 2 - 高频根因识别（是产品缺陷 / 文档缺失 / 用户误操作）
  Step 3 - 可拦截率评估（哪些问题加一条 FAQ 就能减少 30% 工单）
  Step 4 - 自动生成 FAQ 草稿
  ↓
输出：
  - 工单分析报告（问题分布饼图 + 根因树）
  - 优先级改进建议（对客服/产品/技术各部门）
  - 自动生成的 FAQ 更新包（可直接上线）
```

### 技术栈
- Python + pandas（数据处理）
- Claude API（分类 + 根因分析 + FAQ 生成）
- Streamlit（交互界面，上传 CSV 就能跑）

### 可行性评估
- **开发难度**：中等（数据处理 + Claude 多步分析）
- **数据来源**：使用脱敏数据 / 模拟数据完全合规
- **7天能完成**：可以，Streamlit 界面 + Claude 分析链路 4天搞定
- **路演亮点**：现场上传一批工单 CSV，实时跑出分析报告，给客服部门指出"你们 40% 的工单可以被这 5 条 FAQ 消灭"

### 风险
- 需要客服部门配合提供脱敏数据（或提前准备模拟数据）
- 分析准确率依赖数据质量

## 方案四：智能招商素材生成器

### 命题方向
招商支持（方向 2）

### 目标段位
**Lv.2（+15%）** — 解决 BD 团队实际工作痛点

### 问题陈述
BD 团队拓展代理商时，每次都要手动定制招商材料，针对不同地区/渠道类型的话术不同，效率极低。

### 系统设计

```
输入：
  - 目标代理商信息（地区、渠道类型、规模、主营业务）
  - 合作模式选择（CPA / Revenue Share / 混合）
  ↓
Claude 生成：
  - 个性化招商 Pitch（突出对该渠道最有价值的卖点）
  - 多语言版本（中/英/日/韩）
  - 代理商利润估算说明（基于输入的预估交易量）
  - 常见顾虑的应对话术（FAQ for BD）
  ↓
输出：完整招商素材包（Markdown / 可导出 PDF）
```

### 技术栈
- Python + Claude API
- Streamlit 界面（表单输入 → 素材生成）
- WeasyPrint（生成 PDF 导出）

### 可行性评估
- **开发难度**：低（主要是 prompt 工程）
- **实际价值**：BD 团队立即可用
- **7天能完成**：非常可以，2天搞定
- **路演亮点**：现场输入一个目标代理商画像，30秒生成完整中英双语招商包

### 风险
- 创新度相对较低，认知突破度评分可能不高
- 建议做成 Lv.3：同时分析当前招商话术的不足，"掀 BD 团队的桌子"

## 方案五：上币评估 AI 系统（强烈推荐）

### 命题方向
开放命题（方向 5）

### 目标段位
**Lv.5（+35%）** — 掀领导的桌子，挑战上币决策逻辑

### 问题陈述
交易所上币是高影响力决策，但现有流程依赖人工调研，主观性强、覆盖面有限、效率低。没有人能快速、系统地评估一个项目"到底值不值得上"。

### 系统设计

```
输入：项目名称 / Token 合约地址
  ↓
Claude 驱动多维度自动调研：

  链上数据层（调用公开 API）：
  - CoinGecko / CoinMarketCap：市值、交易量、持币分布
  - DeFiLlama：TVL 趋势
  - 合约审计报告（Certik / Slowmist 公开数据）

  社区与舆情层（爬取公开信息）：
  - Twitter/X 提及量 & 情绪分析
  - Reddit 讨论热度
  - 官方文档完整性评估

  竞品对比层：
  - 同赛道项目对比（是否已在 Binance/OKX 上线）
  - 差异化优势分析

  风险评估层（Claude 分析）：
  - 团队背景与历史记录
  - 代码开源情况
  - 监管风险信号

  ↓
Claude 综合评分 + 报告生成：
  - 上币推荐度：强烈推荐 / 建议观望 / 不建议（附理由）
  - 各维度评分雷达图
  - 核心风险警示
  - 推荐上线时机与条件
```

### 技术栈
- Python + 多个公开 API（CoinGecko、CoinMarketCap、Etherscan 等，均免费）
- Claude API（综合分析 + 报告生成）
- Streamlit 界面
- Plotly（雷达图可视化）

### 可行性评估
- **开发难度**：中高（多数据源聚合 + Claude 复杂分析链）
- **数据来源**：全部公开数据，合规无风险
- **7天能完成**：有挑战但可行，建议第一天确定好 API 接口
- **路演亮点**：现场输入一个热门项目代币名，3分钟内生成完整评估报告，直接对比人工调研结论，展示 AI 能替代 3 天的人工调研
- **认知突破度**：★★★★★ — "上币决策可以被 AI 系统化" 会让所有人说没想到

### 风险
- 数据 API 有速率限制，需要合理设计缓存
- Claude 的评估结论准确性依赖 prompt 质量，需要大量调试
- Lv.5 需要在路演时"挑战领导决策"，要有底气展示结论可信度

## 最终建议

**如果只选一个，选方案五（上币评估 AI）。**

理由：
1. 段位最高（+35%）
2. 加密交易所上币是核心业务，AI 化这个流程认知突破度极强
3. 全部用公开数据，完全合规
4. Claude 在这里是真正的核心引擎（不只是翻译）
5. 路演时"现场输入 → 3分钟出报告"，视觉冲击力最强

**如果 7 天时间紧，选方案三（客服工单诊断）作为备选。**
Streamlit + CSV上传 的开发量最可控，且跨部门视角保证 Lv.4 加分。
